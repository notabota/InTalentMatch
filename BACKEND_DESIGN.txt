Backend Design


Conventions

The backend is the server-side half of the same Next.js project that ships the frontend. Route handlers under src/app/api expose the public HTTP surface; business logic is under src/lib. Everything is TypeScript over Prisma 6 over PostgreSQL 17.

Two layers:

  1.  Thin (src/app/api). Route handlers parse requests, call services, shape responses.
  2.  Thick (src/lib). Services own database access; helpers handle sessions, tokens, email, password hashing, error handling and request parsing.

Route handlers depend on services; services depend on Prisma; nothing else depends on a route handler.

2. Data Model

  Fifteen tables, organised in eight functional areas. Column names follow PascalCase on disk through @map/@@map and camelCase in code, so the SQL surface stays self-documenting.

  Identity is a single AspNetUsers table holding the account, password hash, security stamp, contact fields and the Identity-table flags.

  Roles are Consumer and Provider, each one-to-one with the account. Consumer carries posted/completed counters; Provider carries description, totalRating, reviewCount and Premium flags.

  Marketplace is Request and Offer with a cyclic foreign key (a Request's selected Offer, an Offer's parent Request) expressed through two named Prisma relations and two independent SQL constraints.

  Reputation is Review (one-to-one with completed Request through a unique constraint). Messaging is Message between two accounts with a composite index on (sender, receiver). Notifications are append-only alerts addressed to an account. Payment splits into PaymentMethod (single-table inheritance, cards today) and PaymentCommand (append-only ledger of debits, credits and transfers). Taxonomy is a flat Category list plus two composite-key join tables.


3. Service Layer

  Routes never talk to Prisma directly; services hold every database call.

  3.1  Category: getCategories, getCategoriesFor (idempotent upsert by name list), getProviderCategories, setProviderCategories, getRequestCategories, setRequestCategories.

  3.2  Profile: Consumer and Provider CRUD plus updateProvider, a callback-style mutator that runs the route's transform inside the same transaction as the load and save. Provider reads call applyLazyExpiry to clear stale Premium flags in place.

  3.3  Request: createRequest, getRequest, getRequests, getCompletedRequests, findRequests, completeRequest (transactional with a beforeCommit callback that receives the transaction client). Offers go through createOffer, getOffer, getOffersForRequest, selectOffer.

  3.4  Review: addReview increments the Provider's totalRating and reviewCount in the same transaction.

  3.5  Message: getMessages, sendMessage. Send also writes a notification to the recipient.

  3.6  Notification: getNotifications, sendNotification. Send writes the row and dispatches a Postmark email when NOTIFICATION_EMAIL=true.

  3.7  Payment: getPaymentMethods, getPaymentMethodById, addCard. Money movements split into transfer (opens its own transaction) and transferInTx (accepts an existing transaction client), letting callers compose multi-step writes atomically.

  3.8  Premium cron: runPremiumExpiry. Returns counts of inactiveExpired, renewed and cancelled.


4. Cross-Cutting Modules

  - prisma.ts: singleton client, cached on globalThis in dev.
  - session.ts: cookie read, returns accountId or throws SessionExpiredError.
  - password.ts: bcrypt-12 hash and verify, eight-character minimum policy.
  - tokens.ts: HS256 JWT sign and verify, anchored to security stamp.
  - email.ts: Postmark client behind one sendEmail entry point.
  - http.ts: ok, badRequest, unauthorized, notFound, withErrorHandler.
  - reqparse.ts: normalises JSON, form-urlencoded and multipart inputs.
  - errors.ts: canonical error strings.
  - mappers.ts: row-to-DTO converters keeping Prisma types out of routes.
  - models.ts: DTO types.
  - premium.ts: calculatePremiumPrice, calculateTransactionFee.


5. Patterns

  1.  Atomic transactions. Every multi-row mutation runs inside prisma.$transaction.
  2.  Composable transactions. transfer / transferInTx let callers run payments alone or compose them inside a wider business transaction.
  3.  Lazy self-correction. Stale Premium flags are cleared on every Provider read; the daily cron handles bulk maintenance.
  4.  Stateless tokens. Confirmation and reset JWTs are anchored to securityStamp; a successful reset rotates the stamp and invalidates outstanding tokens with no token table to maintain.
  5.  Cookie session. The cookie IS the session record; logout deletes it.
  6.  Side-effect notifications. Notifications are server-write only; clients can only read.


6. Migrations, Seed and Background

  Schema changes happen in prisma/schema.prisma with the matching SQL migration generated by prisma migrate dev. prisma migrate deploy applies pending migrations in CI before traffic switches. Migrations are backwards-compatible with the deployed code revision by policy; rename and drop changes span two releases.

  The seed (prisma/seed.ts, executed by tsx) loads three accounts, seven categories, payment methods, requests, offers, notifications and messages.

  The Premium sweep at POST /api/cron/premium authenticates against CRON_SECRET, downgrades cancelled subscriptions and attempts renewal debits on active ones.


7. Configuration

  DATABASE_URL          Prisma connection.
  AUTH_SECRET           ≥ 32 characters; signs JWTs.
  APP_ORIGIN            Public URL for email links.
  POSTMARK_API_KEY      Postmark API key.
  POSTMARK_FROM         Sender identity.
  CRON_SECRET           Bearer secret for cron endpoints.
  NOTIFICATION_EMAIL    "true" sends email copies of notifications.
