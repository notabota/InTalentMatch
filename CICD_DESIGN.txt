CI/CD Design


Conventions

CI/CD lives entirely in GitHub. Source on GitHub, automation as GitHub Actions, secrets in GitHub Environments, deployment to Vercel. Every commit triggers Continuous Integration; commits on main trigger Continuous Deployment to staging; tagged releases promote staging to production.

One Next.js bundle, one runtime image, one Postgres database per environment. The build artefact CI produces is the same artefact every environment runs.


1. Branch Model

  Main is the single trunk, always deployable, squash-only merges. Every feature ships through a Pull Request.

  Topic branches are named feature/<short>, fix/<short> or chore/<short>, created from main and deleted on merge. Production deploys are gated on annotated SemVer git tags (v0.4.2, v0.5.0); pushing a tag triggers the promotion workflow.


2. Workflow Files

  .github/workflows/ holds four files, each scoped to a clear trigger.

  File                      Trigger                            Purpose
  ------------------------  ---------------------------------  ----------------------------------------
  ci.yml                    pull_request, push to main         Run all CI gates
  deploy-staging.yml        push to main (after CI passes)     Deploy artifact + migrate staging
  deploy-production.yml     tag v* or workflow_dispatch        Promote staging artifact to production
  pr-checks.yml             pull_request                       Title format, labels, stale comments


3. CI Gates

  Every PR runs the same checks in parallel.

  1.  Lint. npm run lint (eslint + eslint-config-next). Errors block merges.
  2.  Typecheck. npx tsc --noEmit against tsconfig.json with noImplicitAny on.
  3.  Build. npm run build emits .next/ as a workflow artifact.
  4.  Test. npm test boots a disposable Postgres through services:, applies prisma migrate deploy, runs unit + integration + contract suites with shard parallelism.
  5.  Migrate-check. npx prisma migrate diff catches schema-without-migration PRs.
  6.  Security audit. npm audit --omit=dev at high severity.

  Every gate is independently runnable locally with the same command the workflow uses.


4. Continuous Deployment

  Staging deploys automatically on push to main once CI passes. The workflow downloads the CI artifact, calls vercel deploy --prebuilt --target=staging, then runs prisma migrate deploy against the staging Postgres using the staging GitHub Environment's DATABASE_URL. A post-deploy smoke probe hits the public read-only surface and asserts 200.

  Production deploys are explicit. They are triggered by pushing a v* tag or by workflow_dispatch from the GitHub UI. The production GitHub Environment requires reviewer approval. The production deploy reuses the artifact already exercised in staging (no rebuild from source). prisma migrate deploy runs against the production Postgres before traffic switches; Vercel's deploy semantics are atomic, so the new revision becomes the production alias only when healthy. A failed smoke probe pages the on-call rotation.


5. Migrations

  Schema changes happen in prisma/schema.prisma with the matching SQL migration generated by prisma migrate dev. CI's migrate-check catches PRs that edit the schema without generating a migration. The integration job applies prisma migrate deploy against a fresh disposable database and re-runs the seed.

  Every migration is backwards-compatible with the deployed code revision. Rename and drop changes span two releases. Ad-hoc data fixes ship as one-off scripts under scripts/fixups/<date>-<purpose>.ts, reviewed in a PR, executed by a designated operator with each run logged.


6. Environments and Secrets

  Three environments:
    - Development. Local Postgres per developer; .env gitignored.
    - Staging. Vercel project + managed Postgres dedicated to staging.
    - Production. Vercel project + managed Postgres dedicated to production.

  Two GitHub Environments (staging, production) hold deploy-time secrets such as DATABASE_URL for the migration step, CRON_SECRET, and integration tokens. Runtime secrets (AUTH_SECRET, POSTMARK_API_KEY) live in Vercel's per-environment settings, so the CI workflow never sees long-lived application secrets and the runtime can rotate them through a dashboard change.


7. Branch Protection

  Branch protection on main enforces:

  - All six CI gates pass before the merge button enables.
  - At least one reviewer approves and may not be the PR author.
  - Squash-only merges keep main's history linear.
  - Force-push is disabled; reverts ship as PRs.
  - Renovate or Dependabot raises PRs for dependency updates; patch and minor versions auto-merge on green CI.


8. Rollback and Observability

  Rolling production back is a one-click Vercel revert (or one CLI call) to the previous immutable deployment. Schema migrations are not auto-reversed; because every migration is backwards-compatible, the previous code revision keeps working against the post-migration schema. A migration-induced regression ships as a forward-fix PR.

  Workflow results post to a Slack channel; red builds on main page the on-call rotation. Vercel logs drain to a long-term store. Sentry captures runtime exceptions with source maps uploaded at build. The cron sweep is asserted hourly by a separate scheduled probe.


9. Workflow Permissions

  Each workflow uses GITHUB_TOKEN with the minimum scopes needed.

  Workflow                     Scopes
  ---------------------------  -------------------------------------------------
  ci.yml                       contents: read
  deploy-staging.yml           contents: read, id-token: write
  deploy-production.yml        contents: read, id-token: write, deployments: write
  pr-checks.yml                pull-requests: write

  No workflow has contents: write, so automation cannot push to the repository. Releases and tags are pushed by humans.
